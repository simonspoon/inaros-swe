# Execute Mesa Task

Glue skill: chains task-pickup → refine → ship → retro → cleanup into one command. `$ARGUMENTS` = task id. Doesn't replace `refine`/`orchestrate`/`retrospective` — invokes them.

## Steps

1. **Fetch** — `mesa task show <id>`. No id given → `mesa task next --project <P>` (P = resolve by repo basename, per `orchestrate` skill's mesa-backbone section). Task missing / already done / blocked → stop, report, don't guess. **Verify the ask against current code before refining** — a task filed days ago may describe a gap that has since been closed (observed, task 443: the requested CLI flag already shipped under an earlier task). Already implemented → don't re-implement, and don't invent a bigger project to justify the ticket. Report it, then either close the task pointing at the commit that did it, or scope down to the *adjacent* gaps the original request implies (parity on a sibling surface, missing test coverage, stale docs) — naming both the pre-existing commit and the residue in the task result so the record shows what was and wasn't new work. **Umbrella task** (`mesa task list --parent <id>` returns a non-empty array of children) → the task is the *batch*, not a unit of work: the parent's acceptance is normally "all children done + some roll-up artifact". Don't refine the parent as if it were one job. Pick each child up (`in_progress`) before dispatch, fan out one agent per child when they're independent, verify each child's deliverables against *its own* acceptance as it lands, then produce the parent's roll-up artifact and close the parent last. Children that depend on each other → refine's ORCHESTRATE route instead of a flat fan-out.
2. **Pick up** — flip `in_progress` immediately. No dispatched engineer here to do this later — this skill IS the pickup.
3. **Refine** — pass the task's title+description+attachments as the raw request to the `refine` skill. Refine restates intent, resolves unknowns, routes INLINE or ORCHESTRATE. This skill does NOT re-interview — refine alone owns `AskUserQuestion`.
4. **Work** — running as a background session → `EnterWorktree` FIRST, before the first `Edit`/`Write` (a direct edit to the shared checkout is rejected and wastes a turn). **Check `git rev-parse --show-toplevel` first: not a git repo → skip `EnterWorktree` entirely** (it will fail), work in place, and skip step 6 — say so in step 11 rather than reporting a commit that can't exist. A non-git working dir is normal for research/scratch trees; it is not a reason to stop. **That check is a snapshot, not a standing verdict — the task itself may create the repo** (a "scaffold a new project" task ending in `git init`). Once it does, the bg-isolation guard starts rejecting edits that were fine minutes earlier, so any *further* edit after a repo appears needs `EnterWorktree` + step 6's ff-merge after all, and step 6 is no longer skippable (observed, task 475: a greenfield MAUI app built in place across 13 stories, then a post-hoc bug fix rejected with "hasn't isolated its changes yet" because the final story had `git init`-ed). Re-check git-ness before editing after any story that could have created a repo. **Task needs edits to a SEPARATE repo nested inside the shared checkout** (a vendored/scratch upstream clone — `realm-src/realm-dotnet`, `third_party/<x>` — typically untracked or gitignored) → the isolation guard rejects those paths too: it is path-based on the shared checkout, and does not care that the nested tree is its own repo. Don't reach for Bash to sidestep the guard. Instead give the nested repo its own worktree *inside* the session worktree: `git -C <nested-repo> worktree add <session-worktree>/<same relative path> <branch>` (detach the original checkout first if it already holds that branch). Edits then land inside the isolation boundary, and commits survive `ExitWorktree` because the branch lives in the nested repo's shared object store — but `git worktree remove` it in step 10 BEFORE exiting, or you leave a stale registration behind (observed, task 498: patching a nested realm-dotnet clone to fix Realm on .NET 10). Then inline pass or full pipeline, per refine's route. Verify per the *plugin's* CLAUDE.md §4 (`/Users/simonspoon/.claude/skills/inaros-swe/CLAUDE.md` — not the working repo's, which may have no numbered sections at all) — test / typecheck / run+observe / diff-vs-criteria, strongest the task allows.
5. **Update documentation** — check whether the work touched behavior a doc describes: README.md, CLAUDE.md, dev docs, user docs. Affected → update in the same pass, same worktree. **Re-anchor every doc path to the worktree before editing.** Repo-root docs (CLAUDE.md, README.md) are typically read at session start — *before* `EnterWorktree` — so the absolute path sitting in context points at the MAIN checkout; reusing it silently edits the shared tree. That dirties main on exactly the files your commit touches, which is what makes the step-6 ff-merge abort. Cheap guard: the path you pass to Edit must start with the worktree root, and `git -C <main> status --short` is clean before step 6 (observed, task 449: CLAUDE.md edited in the main checkout, caught only by the pre-merge status check). Nothing affected → skip, don't manufacture an edit. **Docs for a tool live outside its repo too**: changing a CLI this plugin wraps (khora, loki, qorvex, mesa, uriel, ...) → grep the matching `skills/<tool>/SKILL.md` for the behavior you changed. A skill teaching the workaround your change just removed is the most expensive kind of stale doc — it keeps prescribing it to every future agent (observed, task 465: `blur` shipped, and khora's SKILL.md still taught `key Tab` as the blur step). Different repo → its own commit there, per (a) in step 9; name it in the step-7 result so the record shows both.
6. **Commit to main** — direct commit, no PR. Matches this task's own instruction and the user's standing git preference (commit direct to main). This supersedes the background-job default of isolate→PR→ask for *this* explicit instruction only — don't silently fall back to opening a PR, and don't generalize the skip to unrelated commits. If step 4 isolated work in a worktree, "direct commit to main" means: commit on the worktree branch, then from the main checkout `git merge --ff-only <worktree-branch>` — never `git fetch . <branch>:main`, which desyncs the checked-out tree. In a background-job session the shell's cwd snaps back to the pinned worktree between Bash calls even after a `cd` to the main checkout succeeds for one call — use `git -C <main-checkout-path>` / `scripts/build.sh` run via an explicit path, not a lingering `cd`, for every main-checkout command in this step (merge, rebuild, install). **Main checkout dirty on files your commit touches** → the ff-merge aborts ("local changes would be overwritten"). Don't `update-ref` past it (moves the ref only; the stale tree then shows your change as an apparent *revert* the user could commit) and don't discard their work. Set only the overlapping paths aside, tagged, and put them straight back: `git -C <main> stash push -m "<task-id>-land" -- <paths>` → capture the SHA via `git stash list --format='%H %gs'` *before* anything else touches the shared stack → `merge --ff-only` → `git stash apply <sha>` (never bare `pop`) → verify both survived (their changes present AND yours committed) → drop the entry by re-finding it by tag. Disjoint lines 3-way merge clean. **Never pipe a build/verify command into `tail`/`head` and read `$?`** — in a pipeline `$?` is the *last stage's* status, so a build that failed reports `EXIT=0` and the `&& cp …` that installs the binary still fires, overwriting a good install with a stale one. Redirect to a file and check the exit code of the command itself: `scripts/build.sh > /tmp/build.log 2>&1; echo "EXIT=$?"; tail -5 /tmp/build.log`. And read the tail you asked for: a trailing `Found 1 error.` is a failure even when the next command succeeds (observed, task 492: a post-merge rebuild failed on a dependency the main checkout hadn't installed, the piped `$?` said 0, and `mesa --version` printed happily off the *old* binary). Comes before step 7: when `<value>` below is the commit SHA, it doesn't exist until this step runs. **Work finished but RED against its own Goal** — the change is complete and correct as far as it goes, yet a verification the Goal named still fails for a reason outside the task (upstream bug, EOL dependency, unavailable host capability) → do NOT auto-commit it to main. Landing a knowingly-broken state on main is exactly the hard-to-reverse, outward-facing move that needs consent, and "commit direct to main" was authorized for *working* changes. Stop and put the choice to the user with concrete options — land on main anyway (documented loudly), branch + draft PR so main stays green, revert, or escalate to fixing the upstream cause — then do what they pick. Record the blocker with evidence (a minimal reproduction outside the repo, and the hypotheses you ruled out) in the repo and in step 7's result, not just in chat (observed, task 497: a .NET 10 retarget where the unit half went 39/39 green but the iOS app crashed at startup on an EOL dependency's native-library lookup; the user chose to land red, but that was theirs to choose). **Git repo, but the work produced zero repo changes** → this step is a legitimate no-op, distinct from the non-git case in step 4: the deliverable lives outside the tree (a mesa storyboard / task record / inbox item, a remote config, a report handed back in prose). Don't manufacture a file to have something to commit, and don't leave step 7's `--artifact` blank waiting for a SHA — point it at the external deliverable instead (`mesa storyboard 14 (project 19)`), and say "no commit — deliverable is X" in step 11 (observed, task 491: "create a mesa ERD for the data models" landed entirely in the mesa DB against a clean checkout).
7. **Complete** — `mesa task update <id> --status done --artifact <value> --result-file -` piping the step-11 summary on stdin (or `--result "<text>"` inline for a short one), so the durable narrative lives on the task itself, not a scratch file (`<value>` = commit SHA/path pointer — `--artifact` stays a pointer, `--result` carries the prose). There is no `mesa task done` subcommand.
8. **Retrospective (report only)** — invoke `retrospective` skill against this session (task 1–7 + this skill's own run): have it emit only its Step 4 findings, skip its Step 5 gate — **this skill's own step 9 replaces that Step 5**. Prose in this file describing the override doesn't reach retrospective's context; only the invocation's own `args` do — pass, verbatim, at the call site: "Producing findings as internal data for a programmatic caller (execute-mesa-task step 9), not a report for a human. Emit only Step 4's findings list. Do not reach Step 5, do not ask for approval, do not address a user, do not invite confirmation." If step 4/5 edited `retrospective/SKILL.md` itself earlier in this run, the Skill tool call here may return stale cached content (observed: a same-session edit didn't show up in the injected text even after merging to main) — verify against disk (`Read`/`git show HEAD:...`) before trusting what got injected.
9. **Auto-apply high-confidence findings** — runs in the same turn as step 8, immediately: the findings list just received is internal input to this step, not a user-facing deliverable — receiving it is not a stopping point, don't end the turn or go idle waiting for a reply before this step runs. Concretely, this means: do NOT echo retrospective's findings list back as a chat message and stop there — that assistant message ending the turn *is* the stall this note warns about, wording elsewhere notwithstanding (observed directly, mesa task 402: the findings were printed as a formatted report and the turn ended, requiring an external "are you done?" nudge to resume — the exact failure task 401 already tried to fix by wording alone). Findings surface to the user exactly once, folded into step 11's summary — never as their own message. This is the one place this skill overrides `retrospective`'s default report-only gate, per this workflow's explicit charter. Do this yourself; never surface `AskUserQuestion` for a finding. High-confidence = cites a concrete moment from this run AND has a low-blast-radius owner:
   - (a) skill wording fix → edit the named `SKILL.md` directly, then commit that edit in its plugin repo — an applied-but-uncommitted skill fix is indistinguishable from unapplied and can linger as stray dirty state.
   - (c) memory fact → write it (standard memory protocol).
   - (d) KB-worthy → run `/kb-capture`.
   - (f) missed KB consult → run `inaros-kb:kb-lookup` retroactively for the named topic; fold any hit into the step-11 result narrative.
   - (e) tool change request → `mesa inbox add`.
   - (b) config / permission / hook change → never auto-apply, even here — `retrospective`'s hard rule survives this override. Print the command for the user.
   No cited moment, or genuinely speculative → leave in the report, don't apply.
   Confidence itself unclear (moment is cited but you're unsure the fix is right, or blast radius is ambiguous) → apply the candidate edit to the named `SKILL.md` in the working tree (uncommitted), then consult guru (Agent tool, `subagent_type: "inaros-swe:guru"`) — pointers only: target `SKILL.md` path, repo path, diff base ref `HEAD`. Never your plan or reasoning. Don't ask the user. AGREE → commit it; OBJECT → revert the working-tree edit, treat like (b): print for the user instead of applying.
10. **Cleanup** — remove this run's temp files **and kill anything step 4 left running** (a dev/API server started to verify against). `kill %1` does NOT work: each Bash call is a fresh non-interactive shell with no job control, so the `kill` silently no-ops and the server outlives the run holding its port — kill by port instead, `lsof -ti :<port> | xargs kill` (observed, task 474: a `mesa serve --port 7791` on a throwaway db survived its own `kill %1` and was caught by guru, not by this step). If a worktree was created for the task, verify the commit landed on `main` FIRST (`git log --oneline main..<worktree-branch>` empty, or `git log -1 main` shows the SHA) — the worktree branch is the only copy of the commit until merged, and `ExitWorktree`/`git branch -d` deletes it irreversibly. Only then `ExitWorktree` (`remove`); `keep` only if the task ended blocked/unfinished.
11. **Summarize** — one short report: task id/title, route taken (INLINE/ORCHESTRATE), what shipped (commit SHA — or the external deliverable plus an explicit "no commit", per step 6), docs updated, findings applied vs left for the user, cleanup done.

## Notes

- Steps 6 and 9 each deliberately break another skill's default (background-job PR-first; retrospective's report-only). Both breaks are scoped to this skill's own explicit charter — don't carry "skip PR" or "auto-apply" into other, unrelated invocations of refine/retrospective.
- Task turns out too large for one inline pass → follow refine's own ORCHESTRATE route. This skill doesn't force inline; it forces nothing beyond the chaining.
